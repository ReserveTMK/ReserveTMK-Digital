# ReserveTMK Digital

## Overview
ReserveTMK Digital is a full-stack web application designed to track and manage mentorship relationships, focusing on measuring mentee growth. It provides tools for contact management, logging interactions, and using AI to analyze conversation transcripts for insights into mindset, skill, and confidence. Key capabilities include voice recording, AI analysis, trend visualization, comprehensive reporting aligned with conditional Tāmaki Ora principles, Google Calendar integration, and a mobile-friendly PWA. The project aims to deliver detailed insights into engagement, delivery, and impact for mentorship organizations and individuals.

## User Preferences
Preferred communication style: Simple, everyday language.

## System Architecture

### Overall Structure
The project utilizes a monorepo approach with `client/` (React frontend), `server/` (Express.js backend), and `shared/` (shared types, schemas, and API contracts).

### Frontend Architecture
The frontend is built with React and TypeScript using Vite, Wouter for routing, TanStack React Query for state management, shadcn/ui for UI components, and Tailwind CSS for styling. Recharts handles data visualizations, and the application functions as a mobile-first PWA.

### Backend Architecture
The backend is an Express.js application with Node.js and TypeScript, exposing a RESTful JSON API. Authentication is managed via Replit Auth (OpenID Connect). Text-based AI functionalities are powered by Anthropic Claude through Replit AI Integrations. Audio features (speech-to-text, text-to-speech, voice chat) and image generation are handled by OpenAI. Server-side audio processing relies on ffmpeg.

### Database
PostgreSQL is used as the database, integrated with Drizzle ORM. The schema includes tables for users, contacts, interactions, meetings, events, impact logs, taxonomy, programmes, bookings, bookable_resources, desk_bookings, gear_bookings, groups, funders (with outcomeFocus and reportingGuidance fields), organisation_profile, projects, and reporting data, utilizing JSONB columns for flexible metric storage.

### Key Design Decisions
1.  **Shared API Contracts**: Zod schemas ensure type safety and consistency.
2.  **Replit Auth Integration**: Streamlines user authentication.
3.  **JSONB for Flexible Metrics**: Allows for schema evolution.
4.  **SSE for Voice Streaming**: Enables real-time audio playback.
5.  **AudioWorklet for Playback**: Provides smooth, low-latency streaming audio.
6.  **Community Lens Filtering**: Implements ethnicity-based audience filtering for reporting.
7.  **NZ Timezone Standardization**: All date and time calculations are standardized to Pacific/Auckland.
8.  **UI Terminology**: Uses "People" for contacts, 12-hour time format, and "Venue Hire" in UI labels.
9.  **Nav Structure**: Top nav groups: Dashboard, Community, Delivery, Tracking, Reporting, Ops, Settings. "Ops" contains Projects, Agreements, Funders. "Settings" contains Availability, About Us, and Team (admin-only). Gmail Import accessed from People page header (not in nav).
10. **Invite-Only Access Control**: Platform is private — only users explicitly invited by the admin can log in. Admin user (userId `54568936`) is hardcoded and always has access. The `allowed_users` table tracks invited emails with status (pending/active/revoked). OIDC login callback checks the allowlist before granting access. The `isAuthenticated` middleware also re-checks allowlist status on every request to enforce revocations. Admin manages team via Settings → Team page (`/settings/team`). Non-invited users see an "access denied" message on the landing page.

### Core Features
-   **Reporting Engine**: Funder-focused reporting across 9 sections, including Reach, Delivery, Impact, and alignment with Tamaki Ora. Supports 9 growth metrics grouped by Personal/Venture/Community, journey stage progression tracking (kakano/tipu/ora), and mentoring focus theme summaries. Includes Cohort Analysis (`/cohort-analysis`) for tracking programme cohorts over time — retention, milestones, stage progressions, growth score changes — with single-cohort and side-by-side comparison views.
-   **Mentoring System**: Comprehensive management with AI debrief insights, journey stage tracking, one-click onboarding from application cards, multi-attendee session creation with Google Calendar sync, inline mentor availability calendar in the Schedule Session dialog, inline session notes (summary + next steps) with "Send to Mentee" email, and mentee cards as full onboarding hubs (editable venture/help-with, why mentoring, stuck on, already tried, focus area picker, frequency dropdown, baseline metrics sliders). Session settings contains only Session Types (onboarding setup removed). Pipeline Kanban view with drag-and-drop status transitions (Discovery/Active/On Hold/Graduated).
-   **Growth Surveys**: Self-assessment surveys sent to mentees via email link. Surveys include 1-10 sliders for 7 growth metrics (Mindset, Skill, Confidence, Business Confidence, Systems, Funding Readiness, Network) plus 3 written reflection questions. Managed from "Growth Surveys" tab in mentoring page and via "Send Growth Survey" button on mentee cards. Public survey page renders slider UI for growth surveys. Completed survey responses automatically update the mentee's contact metrics. Schema reuses `surveys` table with `surveyType: "growth"` and `relatedId` storing the mentoring relationship ID. Key files: `growth-surveys-tab.tsx`, `public-survey.tsx` (SliderInput component), `server/routes.ts` (growth survey routes), `server/email.ts` (sendGrowthSurveyEmail).
-   **Scheduling System**: Manages mentor/staff availability with Google Calendar sync and a public booking page for pathway-driven mentee scheduling. Both pathways (mentoring + meeting/hui) use an email-first flow: email → lookup → known contacts skip name step, unknown contacts provide name+phone. Mentoring: new mentees go through goals → discovery session (60 min); returning mentees go straight to session type selection (Quick 30 min / Standard 60 min). Meeting/hui: known contacts go to session type/date; unknown provide name first. Notes are optional via "Add hui notes" toggle. Preview mode (`?preview=true`) shows floating step navigator for admin testing. Confirmation screen shows location/directions from org profile. Calendar events include location + directions in description and send invites to mentees.
-   **Calendar as Tracking Hub**: Integrates Google Calendar sync, manual activity logging, and daily/monthly foot traffic input.
-   **Community Structure**: A 4-tier system (All Contacts / Our Community / Our Innovators / VIP) with dynamic filtering, contact roles, and group management. Groups have 13 types (Business, Social Enterprise, Creative / Arts, Community Organisation, Iwi / Hapū, Government / Council, Education / Training, Health / Social Services, Funder, Corporate / Sponsor, Resident Company, NGO, Uncategorised) and engagement levels (Active/Occasional/Dormant). AI bulk recategorise tool available via edit mode. Gmail import defaults groups to "Uncategorised". Group associations support parent/child/peer relationship types via `relationshipType` column on `group_associations` table (stored directionally: parent stores groupId=parent, associatedGroupId=child). Ecosystem page has Categories/Connections view toggle. Connection management panel accessible from group card menu. When a child group is linked to an impact log, parent groups are auto-tagged. Booking group names show parent as secondary label (e.g., "Te Oro · Auckland Council").
-   **Debriefs/Impact Tracking**: Queue for events needing debriefs, automated AI analysis of interactions, user review, and action item suggestions.
-   **Spaces Hub**: Unified hub at `/spaces` with 5 tabs: Calendar (venue + desk availability grid), Venue Hire (booking management with calendar/kanban/list views — monthly calendar is default), Hot Desking (desk availability cards), Resources (manage venues and desks with two sub-tabs: Venues and Desks), Bookers (regular bookers filtered to venue hire + hot desking categories). Spaces calendar supports deep-linking via URL params (`?date=YYYY-MM-DD&view=week`) using `useSearch` from wouter. Confirming a booking creates a Google Calendar event (venue name, booker, times, location) and invites the booker if they have an email on file. Bookings table has `googleCalendarEventId` column. Venue Hire Settings (gear icon) contains: Reminders, Survey, Portal, Xero. Resource management components extracted to `client/src/components/spaces/resources-tab.tsx`. Each venue card is expandable with tabbed sections for Details (name/description/capacity) and Availability Schedule (per-day operating hours). "Booker Access Info" is managed at the location (spaceName) level — per-location operational instructions (Access, Opening Procedure, Closing Procedure, Emergency) for bookers, displayed as a collapsible section under each location header in the Venues sub-tab. The `venue_instructions` table has instruction types: access, opening, closing, emergency (migrated from old arrival/departure/general types). The About Us page has "Location Instructions" — per-location visitor info (How to find us, Parking, General info) stored as structured JSON in the `venue_directions` column of `organisation_profile`. The Desks sub-tab contains desk resources and hot desking hours. Booking reminders send booker access info before venue hire bookings. The `bookings` table has a `locationAccess` text array column for per-booking location access control. On the booking detail view, admins can select which locations a booker has access to via checkboxes, and confirmed/completed bookings show booker access info. The `booking_reminder_settings` table stores the enabled toggle and send timing. Bookings support multi-venue selection via `venueIds` integer array column (with legacy `venueId` kept for backwards compat). The booking form uses checkboxes instead of a dropdown, so users can tick multiple venues for a single booking.
-   **Gear Management**: Standalone page with 3 tabs: Availability (equipment status cards + active checkouts + mark-returned flow), Inventory (add/edit/delete gear items), Bookers (regular bookers filtered to gear category). Gear inventory management removed from Spaces Resources tab.
-   **Booking Categories**: Unified framework for Venue Hire, Hot Desking, and Gear Booking, managed via bookable resources and linked agreements. Agreements (Memberships/MOUs) have category checkboxes for Venue Hire, Hot Desking, and Gear. Hot desking hours are driven by the operating hours settings in the Desks sub-tab. Booking reminders send booker access info before venue hire bookings.
-   **Regular Booker Portal**: Self-service portal with magic link authentication, showing category-specific usage, subscription status, and booking flows. Regular Bookers page (`regular-bookers.tsx`) supports `embedded` and `categoryScope` props for embedding in Spaces and Gear hubs. Standalone `/regular-bookers` route redirects to `/spaces?tab=bookers`. Sidebar link removed.
-   **Casual Venue Hire**: Public page at `/casual-hire` for ad-hoc/one-off bookers (no login required). Two-step flow: contact info (name, email, phone, optional organisation) → calendar availability + venue selection + booking details. Creates booking as "enquiry" status with `bookingSource: "casual"`. Matches existing contacts by email or creates new ones. Public API endpoints at `/api/public/casual-hire/*` for venues, availability, and booking submission. Availability logic applies — conflicts with existing bookings are rejected. Pricing confirmed by admin after review.
-   **Public Programme Registration**: Allows public registration for programmes with auto-generated links and QR codes, with integrated registration management and attendance tracking.
-   **Funder Profile AI Generation**: AI Generate button on funder edit form uses Anthropic Claude to build out the full profile from existing info + uploaded documents. Generates outcome focus areas with indicators, reporting guidance with rhythms, narrative style, priority sections, and a partnership strategy section describing how to deliver and report to the funder. Schema field: `partnershipStrategy` on funders table.

## External Dependencies

### Required Services
-   **PostgreSQL Database**: Primary data persistence.
-   **Anthropic Claude API** (via Replit AI Integrations): Utilized for text-based AI features like interaction analysis and task extraction.
-   **OpenAI API** (via Replit AI Integrations): Used for audio functionalities (speech-to-text, text-to-speech, voice chat) and image generation.
-   **Replit Auth (OIDC)**: Handles user authentication and authorization.
-   **Xero Integration**: OAuth2 connection for automated invoice generation and contact/invoice syncing.
-   **Gmail Sync**: Multi-account Gmail import with filtering and mailing list detection. Features account selection checkboxes, preview-before-import step, email interaction logging (creates "Email" type interactions with subject lines), duplicate name detection with skip/create/merge options, existing contact org linking, per-account scan buttons, and email context capture (recent subjects displayed in preview and contact notes).

### System Dependencies
-   **ffmpeg**: Essential for server-side audio format conversion.
